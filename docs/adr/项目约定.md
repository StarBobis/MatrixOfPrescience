- 不符合核心思想的给我报告风险点

# 群聊 @ 路由约定
- Vue I18n 文案中的字面量 `@` 必须写成 `{'@'}`，禁止直接写裸 `@名称`，否则会被解析为 linked message 并在运行时中断消息调度。
- 讨论式群聊中，明确 `@` 的成员先回复，未被点名成员可以进入系统控制的观察与共识流程。
- 任务式群聊中，未 `@` 或 `@管理员` 都必须启动完整任务流程：管理员拆解和分派，执行群友完成任务，管理员审查并面向用户汇总；管理员不得代替执行群友完成具体任务。
- 任务式群聊中，仅 `@执行群友` 时只调度被点名的执行群友。未被点名成员不得收到回复决策、投票或生成请求。
- `@管理员` 与执行群友同时出现时仍由管理员启动完整任务流程，避免绕过管理员的统一编排。
- 任务式身份由系统字段 `isAdmin` 决定，不由模型自行判断。只有管理员能输出任务分派；执行群友只能完成自己的任务，禁止继续分派或指挥其他成员。
- 管理员审查输出的 `REDISPATCH` 是系统控制标记。系统必须兼容 Markdown 强调和中英文冒号，提取其后的修正意见并显式注入下一轮管理员规划提示，不能只依赖模型查看历史自行领会。
- 管理员规划属于任务状态机控制输出，不得套用普通聊天的“必须以 `@用户` 开头”规则。计划必须包含至少一项使用真实群友名的非空任务；纠错提示必须使用真实群友名，连续无效时明确报错并停止，禁止显示成功后静默结束。
- 管理员按任务需要选择执行群友，不要求每轮覆盖全部成员。系统只调度计划中具有非空 `- @真实群友名: 任务` 行的成员，发言队列必须与实际调度集合一致。
- `REDISPATCH:` 是否触发与标记后的文本是否为空必须分开判断；即使标记位于审查正文末尾也要进入下一轮。管理员应在重派内容中用 `- @真实群友名: 修正任务` 显式点名下一轮执行者。

# Rust代码规范

- mod.rs中不允许出现除了pub 声明之外的其它实际代码块，实际代码块应该分模块化各自位于自己独立的.rs文件中，由mod.rs负责向外暴漏
- lib.rs应该尽量维持简洁，功能代码应该模块化放到各个子文件夹下对应rs中而不是堆在一起放在lib.rs中
- 除了Command层外，不允许散落的函数，必须以struct + impl组织起来，方便调用时区分作用域
