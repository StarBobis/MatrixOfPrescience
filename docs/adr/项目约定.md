在接下来的对话中遵循如下约定

# 核心思想

- 阅读代码优先使用 CodeGraph，注意及时更新防止内容过期。
- 模块化代码，前端功能要按组件拆分独立 `.ts`、`.vue`，防止单文件过大。
- 代码和注释要写纯英文。
- 改代码别忘了同步改 i18n，注意编码问题，不要引入乱码；如果出现要及时修复。
- 每次修改完代码后，基于 `git diff` 给出一段简短的中文描述，方便提交代码使用。
- 本项目使用 `bun`、`vite` 作为前端包管理与构建工具，不要改用 `npm`。
- 后端应该模块化，可复用内容尽可能使用 `struct + impl` 做成工具类型。
- 不符合核心思想的实现要明确报告风险点。

# 群聊 @ 路由约定

- Vue I18n 文案里的字面量 `@` 必须写成 `{'@'}`，禁止直接写 `@名称`，否则会被解析成 linked message 并在运行时打断消息调度。
- 讨论式群聊里，明确 `@` 的成员优先回复；未被点名成员可以进入系统控制的观察与共识流程。
- 任务式群聊里，未 `@` 或 `@管理员` 时都必须启动完整任务流程：管理员拆解和分派，执行群友完成任务，管理员审查并面向用户汇总；管理员不得代替执行群友完成具体任务。
- 任务式群聊里，仅 `@执行群友` 时只调度被点名的执行群友。未被点名成员不得收到回复决策、投票或生成请求。
- `@管理员` 与执行群友同时出现时，仍由管理员启动完整任务流程，避免绕过管理员的统一编排。
- 任务式身份由系统字段 `isAdmin` 决定，不由模型自行判断。只有管理员能输出任务分派；执行群友只能完成自己的任务，禁止继续分派或指挥其他成员。
- 管理员审查输出里的 `REDISPATCH` 是系统控制标记。系统必须兼容 Markdown 强调和中英文冒号，提取其后的修正意见并显式注入下一轮管理员规划提示，不能只依赖模型查看历史自行领会。
- 管理员规划属于任务状态机控制输出，不得套用普通聊天里“必须以 `@用户` 开头”的规则。规划必须包含至少一项使用真实群友名的非空任务；纠错提示也必须使用真实群友名，连续无效时要明确报错并停止，不能伪装成完成。
- 管理员按任务需要选择执行群友，不要求每轮覆盖全部成员。系统只调度计划中具有非空 `- @真实群友名: 任务` 行的成员，发言队列必须与实际调度集合一致。
- `REDISPATCH:` 是否触发，与标记后的文本是否为空，必须分开判断；即使标记位于审查正文末尾也要进入下一轮。管理员应在重派内容中用 `- @真实群友名: 修正任务` 显式点名下一轮执行者。

# 执行群友输出契约

- 任务式中，执行群友的回复是交付物，不是路由动作。由系统决定下一位发言者，执行群友不应为了“交接”而称呼管理员、用户或其他群友。
- 如果执行群友输出 `- @名称: ...` 这类分派语法，或出现重新分工、指挥别人、把任务交回管理员等行为，系统必须把它视为无效执行，不能当作完成。
- 无效执行需要明确重试一次；如果重试后仍在分派或回派，系统要留下对管理员可见的失败记录，而不是静默标记为完成。

# Rust 代码规范

- `mod.rs` 中不允许出现除了 `pub` 声明之外的实际代码块，实际代码应模块化后放入各自独立的 `.rs` 文件中，由 `mod.rs` 负责对外暴露。
- `lib.rs` 应尽量保持简洁，功能代码应拆分到对应子文件或子目录中，而不是堆在 `lib.rs` 里。
- 除了 Command 层外，不允许散落的函数，尽量使用 `struct + impl` 组织代码，方便区分作用域。
- 工具类、可复用方法应该放到src\utils下，并且以xxx_utils.rs风格命名。
